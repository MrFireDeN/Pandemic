# Pandemic

Веб-версия настольной игры **Pandemic**.
Проект представляет собой серверную реализацию игровой логики с веб-интерфейсом и поддержкой real-time взаимодействия между игроками.

Это **игра**, а не классическое CRUD-приложение.
Основное состояние партии живёт в памяти сервера, база данных используется как механизм персистентности и восстановления состояния.

---

## 1. Что это за проект

Проект реализует:

* создание игровых сессий (host),
* подключение игроков,
* базовый игровой цикл Pandemic,
* синхронизацию состояния между клиентами через Socket.IO,
* сохранение и загрузку состояния игры из SQLite.

Проект **не является финальной реализацией всех правил Pandemic**.
Часть механик упрощена или реализована в прототипном виде.

Цель проекта — **рабочий игровой сервер**, а не академически чистая архитектура.

## 2. Tech stack

**Backend**

* Python
* Flask
* Flask-SocketIO
* SQLAlchemy
* SQLite

**Frontend**

* Flask templates (HTML)
* JS (Socket.IO client)
* CSS

**Архитектурно**

* доменная логика игры вынесена из веб-слоя,
* используется паттерн Observer для игровых событий,
* используется репозиторий для сохранения/загрузки состояния игры.

## 4. Как запустить проект

1. Создать виртуальное окружение и установить зависимости.
2. Убедиться, что доступен SQLite.
3. При необходимости:

    * создать БД (`create_all`)
    * заполнить справочники (города, карты)
4. Запустить сервер:

   ```
   python app.py
   ```
5. Открыть в браузере:

    * `/` — стартовая страница
    * `/host/<code>` — страница хоста
    * `/join` — подключение игрока
    * `/board/<code>` — доска
    * `/player/<code>/<name>` — интерфейс игрока

По умолчанию используется SQLite:

```
sqlite:///pandemic.db
```

## 5. Текущий статус игрового функционала

### Реализовано

* создание и хранение игровых сессий,
* добавление игроков,
* старт игры (раздача карт, стартовая станция, порядок хода),
* перемещение игроков,
* лечение городов,
* постройка исследовательских станций,
* обмен картами,
* открытие лекарств,
* завершение хода (добор карт, заражение, эпидемии),
* базовая проверка поражения/победы,
* сохранение и загрузка состояния игры.

### Частично реализовано

* эпидемии (есть логика, но с багами),
* вспышки (есть защита от циклов, но жизненный цикл visited не завершён),
* real-time события (часть событий заглушена).

### Не реализовано / заглушки

* большинство REST endpoint’ов `/api/player/*`,
* большая часть Socket.IO событий,
* полноценная синхронизация клиента с сервером по всем фазам хода.
* Front практически отсутствует

## 6. Архитектура (как это реально работает)

### Веб-слой

* `app.py` — инициализация Flask, страницы, запуск сервера.
* `controller.py` — REST и Socket.IO точки входа.
* UI работает через Socket.IO комнаты, привязанные к `game_code`.

### Домен

* `PandemicGame` — центральный объект игры.
* `data/*` — чистая игровая логика:

    * города,
    * игроки,
    * колоды,
    * доска.

ORM-модели **не используются** внутри домена напрямую.

### Персистентность

* `GameRepository` отвечает за:

    * сохранение состояния игры в БД,
    * загрузку состояния игры из БД.
* Домен сам инициирует сохранение (`__save()`).

## 7. API и события (кратко)

### REST

* `GET /api/health` — проверка сервера
* `POST /api/host/create` — создать игровую сессию
* `POST /api/player/move` — перемещение игрока
  ⚠️ Сейчас имеет рассинхронизацию с доменной логикой

Остальные REST-эндпоинты существуют как заглушки.

### Socket.IO

* `host_join`
* `player_join`
* `start_game`
* `/game/player:move`

Большинство server-originated событий (`epidemic`, `outbreak`, `game_over`) объявлены, но не реализованы.

## 8. Модель хранения состояния

Важно понимать **двойственность состояния**:

* **Истина игры** — в памяти (`SESSIONS`, `PandemicGame`).
* **База данных** — снапшот состояния, позволяющий:

    * пережить перезапуск сервера,
    * восстановить игру.

ORM-модели и доменные сущности — **разные объекты**:

* `PlayerModel` ≠ `PlayerGame`
* `CityModel` ≠ `CityGame`

Любая попытка смешать их напрямую — архитектурная ошибка.

## 9. Известные проблемы и технический долг (обязательно к прочтению)

### ❗ Рассинхронизация controller ↔ domain

* В `controller.py` используются вызовы и сигнатуры, которых нет или которые отличаются в `PandemicGame`.
* REST и Socket.IO используют разные пути управления игрой.

**Истина — в `game.py` и `data/*`.**

---

### ⚠️ Упрощения и “чит-логика”

* Если ни одно правило перемещения не подходит — используется `cheat move`.
* Колода игроков работает как “карусель”, а не как классическая колода.
* При вытягивании карты с владельцем игра может завершиться.

Это сделано **осознанно для прототипа**.

---

### ⚠️ Эпидемии и заражение

* Infection rate считается не по каноническим правилам.
* Механизм visited-городов при вспышках не всегда очищается.

---

### ⚠️ Загрузка игр при старте сервера

При запуске сервера:

* все сессии загружаются из БД,
* и **сразу сохраняются обратно**.

Это увеличивает время старта и может иметь побочные эффекты при большом числе игр.
